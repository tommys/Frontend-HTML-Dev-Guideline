Introduction

This document contains the guidelines and best practices for the frontend web development team at eLab, A.S Watson.

Each note of detail here represents either:

  • A reminder to follow existing standards or industry conventions.
  • Keep code consistency across our projects.
  • Facilitate ease of maintenance.
  • Ensure we create professional quality websites.
  • Guide staff on-boarding or instruct new developers and digital agency.
  • We know that the same convention for all projects, clients, and teams isnâ€™t practical. For these reasons improvement of this document are encouraged for various projects and teams.
  • What works well can be submitted and discussed as new recommendations moving forward.

HTML Standards and Browser Support

All markup will be written using the latest HTML5 markup specifications from the W3C, as implemented by browsers and devices that meet project requirements.

All HTML, CSS and Javascript are well tested across all major browser (Chrome, Firefox, Safari and IE8+), IOS and Android Devices.

General Development Guide

  • CSS file will be better to be divided by section, global.css (e.g. header, footer, components across site), product.cssâ€¦etc.
  • For all developed HTML template, please use standalones HTML without any includes.
  • For all HTML template, please complied with W3C web standards.
  • Ownership for third party Plugin and Library please make sure they are legal permissions which have been obtained for the use of the material (if any are required).
  • Use relative paths for the HTML template development.
  • Provide fonts file if there are any custom CSS font usages, if they are license required please advise if purchase needed. If license applies please check point #4.
  • Do not add any unnecessary inline html code into js coding to render result in HTML.
  • Avoid inline Javascript and CSS in HTML.
  • From third party plugins or library resources, make sure to save all files (JS, CSS or images etcâ€¦) to local folder. Do not use any external link. If license applies please check point #4.
  • Tables should not be used for page layout; only use them when you need to display tabular data.
  • Use CSS Media Queries to develop Responsive layout; Disable Responsive Layout for IE8 or below browser (For IE8 or below browser uses default preset maximum screen resolution only.)
  • Do not minified or encrypted source code. Except Plugin/Library from third party please use latest stable minified version.
  • Do not keep any unused file(s) in the delivered folder.
  • Load the least amount of JavaScript that is needed. Wherever possible include script files at the end of the HTML document just before the </body> tag.
  • For CMS usage, do not use SVG for any icon or image.
  • Use Icon Fonts or Image Sprites for all small icons (Non-CMS icon only)
  • Should not have duplicate IDs in same HTML page
  • Write efficient and best practices Selector rules please referenace following Link
  • Do not use standalone full set of CSS for different languages (e.g. Chinese and English), create and overrite only necessary changes base on default language.

Responsive Development Guide

  • Do not duplicate 2 set of HTML for responsive coding.
    (Desktop Nav and Mobile Nav must be same set of coding.)
  • Percentage-based grids
  • Flexible images that scale
  • CSS Media Queries
  • Code with global styles that apply across all breakpoints
  • Be careful of images. Do not render a 5MB image at 100px by 100px. This does not change the bandwidth needed to download the file!
  • Test on real hardware. Actual testing on actual devices cannot be substituted with resizing a browser window, or even an "emulation" mode offered by a desktop browser.
  • Review site features for mobile-specific JavaScript events (e.g. touch, pointer, vs. mouse) and interactions (e.g. swipe) on actual mobile devices.

General Formatting Rules For HTML, CSS and JS

Write Well Structured HTML

Not Recommended:

<div class="container">
  <div class="article">
    <div class="headline">Headlines Across the World</div>
  </div>
</div>

Recommended:

<div class="container">
  <article>
    <h1>Headlines Across the World</h1>
  </article>
</div>

Write Valid Semantic Markup

Not Recommended:

<span class="sectionHeading">A Heading</span>
<br /> <br />
Lorem ipsum dolor sit amet. ...
<br /> <br />

Recommended:

<h2>A Heading</h2>
<p>
Lorem ipsum dolor sit amet. ...
</p>

Capitalization

Use only lowercase.

All code has to be lowercase. This applies to HTML element names, attributes, attribute values (unless text/CDATA).

Not Recommended:

<A HREF="/">Home</A>

Recommended:

<a href="#">Home</a>

Trailing Whitespace

Remove trailing white spaces.

Trailing white spaces are unnecessary and can complicate diffs.

Not Recommended:

<p>What?</p>__

Recommended:

<p>What?</p>

Indentation

Indentation should be consistent throughout the entire file.

For all languages, indent your code with tabs. The default tab size should be set as 4.

Not Recommended:

Use space as tab.

Recommended:

Default tab size should be set as 4

Encoding

Use UTF-8 (no BOM).

Make sure your editor uses UTF-8 as character encoding, without a byte order mark. Specify the encoding in HTML templates and documents with <meta charset="utf-8">.

Comments

Explain code as needed, where possible.

Use comments to explain code: What does it cover, what purpose does it serve, and why is the respective solution used or preferred?

Recommended:

It's very important to explain every JS Function does.

HTML Semantics

Use HTML according to its purpose.

Use elements for what they have been created for. For example, use heading elements for headings, p elements for paragraphs, a elements for anchor, etc. Using HTML according to its purpose is important for accessibility, reuse and code efficiency reasons.

Not Recommended:

<div onclick="goToRecommendations();">All recommendations</div>

Recommended:

<a href="recommendations.html">All recommendations</a>

Paragraphs

Avoid using <br /> tags to separate paragraphs or lines of text. Use <p> instead with proper opening and closing elements.

Not Recommended:

Paragraph with text<br />Paragraph with text<br />

Recommended:

<p>Paragraph with text</p>
<p>Paragraph with text</p>

Multimedia Fallback

Provide alternative contents for multimedia.

For multimedia, such as images, video, or animated objects via canvas, make sure to offer alternative access. For images that means use of meaningful alternative text and for video and audio transcripts and captions, if available.

Providing alternative contents is important for accessibility reasons. A blind user has few cues to tell what an image is about without the alt attributes, and other users may have no way of understanding what video or audio contents are about either.

For images whose alt attributes would introduce redundancy and for images whose purpose is purely decorative which you cannot immediately use CSS for, use no alternative text, as in alt="".

Not Recommended:

<img src="watsons.png">

Recommended:

<img src="watsons.png" alt="Watsons logo" />

HTML Quotation Marks

When quoting attribute values, use double quotation marks.

Not Recommended:

<a href='login/' class='btn'>Login</a>

Recommended:

<a href="login/" class="btn">Login</a>

Attribute should not be empty

Not Recommended:

<a href="">Link</a>

			<div class="">content</div>

Recommended:

<a href="javascript:;">Link</a>

			<div class="classname">content</div>
